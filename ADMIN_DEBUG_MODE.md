# 管理员模式说明

## 目标

本次新增的管理员模式，目标不是替代完整自动回归测试，而是让房主可以在真实对局界面里快速构造测试场景，缩短手动验证路径。

核心思路：

- 服务端仍然是唯一权威状态来源。
- 房主通过统一的 `debugCommand` 通道下发管理员指令。
- 客户端管理员面板尽量走“通用目录 + 当前游戏状态”的方式生成，不为单张卡、单个格子写死特殊入口。

这样做的结果是：

- 可以保留接近真实游玩的测试体验。
- 改动面更小，不需要单独搭一整套脚本化测试场景系统。
- 后续新增“同类型内容”时，管理员面板大多能自动覆盖。

## 已实现能力

管理员模式仅房主可用，且在服务端进行权限校验。

当前支持：

- 直接给指定玩家发放任意技能卡
- 直接触发任意事件卡
  - 好运
  - 厄运
  - 彩票
  - 游乐
- 直接把指定玩家传送到任意格子
  - 可选是否立即结算该格子效果
- 直接设置玩家资源
  - 现金
  - 彩券
  - 存款
  - 贷款本金
  - 未付利息
- 直接设置地产状态
  - 归属玩家
  - 等级
  - 是否抵押
- 直接设置玩家股票持仓
- 直接设置股票价格
- 直接设置玩家拘留状态
  - 无拘留
  - 监狱
  - 医院
- 直接设置回合状态
  - 当前行动玩家
  - 阶段
  - 回合数
  - 已完成行动数
  - 是否清空待处理动作

## 使用方式

进入游戏后，只有房主能看到“管理员测试”按钮。

打开面板后：

1. 先选择目标玩家。
2. 再根据要测试的玩法，选择对应区域操作。
3. 每次操作都通过服务端立即写入游戏状态，并像正常玩法一样广播给所有客户端。

建议手测流程：

1. 用“玩家资源”快速补足金币、彩券或银行数据。
2. 用“技能卡发放”或“事件卡触发”直接进入目标内容。
3. 用“传送到格子”或“地产状态”构造地租、升级、抵押等场景。
4. 用“玩家持股”和“股票价格”构造股票收益、亏损、委托冲突等场景。
5. 用“拘留状态”和“回合与阶段”收尾测试特殊流程。

## 架构落点

### 1. 共享协议

在 `shared/src/index.ts` 中新增了管理员模式相关共享类型：

- `DebugCatalog`
- `DebugCommand`
- `DebugCommandAck`

这样前后端都使用同一套协议，符合项目共享类型集中管理的规范。

### 2. 服务端执行器

新增 `server/src/game/debug.ts`，负责解释管理员指令并直接修改权威游戏状态。

它处理两类能力：

- 纯状态改写
  - 资源
  - 地产
  - 持仓
  - 股价
  - 拘留
  - 回合
- 复用现有玩法逻辑的即时结算
  - 指定事件卡触发
  - 指定格子传送并可选结算

### 3. Socket 入口

在 `server/src/socket.ts` 中新增 `debugCommand` 事件。

处理规则：

- 必须先加入房间
- 必须是房主
- 改动状态的操作必须已有游戏实例
- 结果仍通过现有广播链路同步
  - 有 `ActionOutcome` 的直接复用 `broadcastOutcome`
  - 纯状态更新则走 `emitGame`

### 4. 客户端面板

新增 `client/src/components/AdminDebugPanel.tsx`。

客户端面板不会硬编码每张技能卡或事件卡，而是动态读取：

- 当前 `game.players`
- 当前 `game.tiles`
- 当前 `game.properties`
- 当前 `game.stocks`
- 服务端返回的 `DebugCatalog`

## 为什么后续新增同类型内容通常不用补入口

这是本次设计最关键的地方。

### 自动继承的内容

以下内容新增后，通常不需要再给管理员模式新增入口：

- 新技能卡
  - 因为面板直接读取 `skillCardTemplates` 生成技能卡目录
- 新事件卡
  - 因为面板直接读取 `luckCards` 生成事件卡目录
- 新地产格
  - 因为面板直接读取当前 `game.tiles` 中的 `property` 格子
- 新股票
  - 因为面板直接读取当前 `game.stocks`
- 新玩家
  - 因为面板直接读取当前 `game.players`

换句话说，只要新内容属于“已有类型的新实例”，管理员模式一般会自动覆盖。

### 仍然需要补管理员支持的情况

如果以后加入的是“全新类型内容”，那就仍然需要补管理员命令和 UI，例如：

- 新的可持有物类型，但不属于技能卡
- 新的独立资源体系
- 新的特殊结算面板
- 新的状态机阶段，但不属于当前已有回合字段

判断标准很简单：

- 如果只是往现有数据表里加新条目，通常不用改管理员面板。
- 如果是新增一类状态结构或新增一套交互机制，就要补一个新的管理员命令。

## 这次实现对后续维护的帮助

为了降低“架构独立性不足导致管理员模式失效”的风险，这次管理员模式刻意依赖更稳定的公共层：

- 技能卡目录来自技能卡模板数据
- 事件卡目录来自事件卡数据
- 地产、股票、玩家来自当前权威游戏状态
- 管理员指令通过统一协议进入服务端

因此后续维护成本主要集中在：

1. 新玩法是否仍然复用现有状态结构
2. 新玩法是否仍然能被现有通用入口表达

只要答案是“能”，管理员模式就能继续跟上。

## 已知边界

- 管理员模式主要服务于高效手测，不负责自动判断“结果是否符合预期”。
- 目前覆盖的是已有主玩法对应的共性入口，不是所有未来未知机制的万能控制台。
- 如果未来新增的是全新资源类型或全新玩法层，需要补充新的 `DebugCommand`。
